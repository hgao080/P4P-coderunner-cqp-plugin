Code Quality Principles

2026-01-31

The Code Quality Princples (CPQ) are intented to capture, in a general, language-neutral, useful way, the many rules, advice, and code conventions typically referred to when talking about "code style".

1. Clear Presentation

Principle
Different elements are easy to recognise and distinguish and the relationships between them are apparent.
Rationale
Clear layout improves our shared understanding by making the individual elements easy to identify and signalling the elements the author considers to be related.
Guidelines
  1. Characters used should be easily recognisable.
  2. Formatting (horizontal and vertical spacing, indentation, brackets and line length) should make it easy to distinguish separate elements.
  3. Placement of elements within a file should highlight the intended relationships.

2. Explanatory Language

Principle
The rationale, intent and meaning of code is explicit.
Rationale
Being explicit in describing the purpose of the code elements helps us understand the author's intention, thus improving understandability.
Guidelines
  1. Documentation and comments within code files should describe rationale, intent and meaning and be clear and unambiguous to the intended reader.
  2. Names of elements (variables, functions, etc.) should be descriptive and contextually appropriate.
  3. Use symbolic constants to convey the meaning of literals.
  4. Each variable should be used for a single purpose.
  5. Language features that are designed to provide supporting information to the reader should be used where helpful.

3. Consistent Code

Principle
Elements that are similar in nature are presented and used in a similar way.
Rationale
Consistency leverages familiarity to reduce the mental effort required to understand the code.
Guidelines
  1. Documentation and notation should be used consistently throughout.
  2. When making decisions about code, be consistent in implementation choices.

4. Used Content

Principle
All elements that are introduced are meaningfully used.
Rationale
Non-contributing code elements require unnecessary mental effort.
Guidelines
  1. All code constructs should be needed in order for execution to be possible.
  2. It should be possible to execute each line of code.
  3. Remove code that is executed but does not affect functionality.

5. Simple Constructs

Principle
Coding constructs are selected to minimise complexity for the intended reader.
Rationale
Code that is perceived by the reader as simple is easier to understand.
Guidelines
  1. Nesting levels of constructs should be minimised.
  2. Flow of control should be easy to follow with few breaks and exceptions.
  3. The size and complexity of expressions should be appropriate for the intended reader.

6. Minimal Duplication

Principle
Code repetition is avoided.
Rationale
Repeated code can be difficult to change because changes need to be made multiple times, there is a risk that not all items are changed and/or difficult to understand because you have to read more of it.
Guidelines
  1. Repeated groups of statements in different places should be consolidated in one place.
  2. Repeated statements next to each other should be replaced by a loop containing the group.

7. Modular Structure

Principle
Related code is grouped together and dependencies between groups minimised.
Rationale
Placing related elements together makes code easier to understand. Reducing inter-connectedness means that isolated pieces can be more easily understood and can be modified independently.
Guidelines
  1. Modules should be created with clearly defined responsibilities and minimal dependence on other modules.
  2. Functions should aim to implement a single task and restrict access to what is needed.
  3. Code should be organised such that conceptually related elements appear together.
  4. The scope of constants and variables should be minimised.

8. Problem Alignment

Principle
Implementation choices are consistent with the problem to be solved.
Rationale
An implementation that reflects the problem is easier to understand and change.
Guidelines
  1. Data structures should align with the data you are representing.
  2. Algorithms should reflect the process they represent.
  3. Code constructs should align with the algorithm.
  4. Modules should align with the problem to be solved.

Publications relating to CQP

CQP has been developed (and is still developing) from a long-term research project. It originally had the limited goal of determining how to help instructors teach what is typically known as "code style".

While the principles have not changed in general, the names, wording of their rationale, and choice of guildlines have change over time as our understanding of what is really intended by the different advice and rules has improved. The biggest change has been changing the name of the frame work from Code Style Model (CSM) to Code Quality Princples (CQP) (See [6] below).
